# ADR-0001：首版采用 local-first 桌面模块化单体

- 状态：Rejected（由 ADR-0004 取代）
- 日期：2026-07-19

## 背景

产品需要直接扫描本地书库、离线阅读、保存进度和处理大型 PDF/EPUB。当前尚未确认多人账号、NAS 或公共 Web 访问是否为首发硬需求。

2026-07-19 用户确认目标为 NAS 上的多用户 Web 应用，因此本 ADR 不再适用。

## 决策

默认采用 Tauri v2 + React/TypeScript + Rust + SQLite 的 local-first 桌面模块化单体。UI 和领域逻辑保持边界，远端同步仅预留 port，不在首版实现。

## 理由

- 本地文件访问、目录监控、离线和隐私是默认场景的主路径。
- Web 技术可以复用 PDF.js、foliate-js 等阅读生态。
- SQLite 和进程内持久化任务能够避免首版部署数据库、队列和身份服务。
- 模块化单体保留未来拆出同步服务的可能，但不会预付微服务复杂度。

## 备选方案

### 自托管 Web：Spring Boot/Angular 或 .NET/Angular

适合家庭 NAS、多用户和跨设备浏览器访问；代价是部署、认证、网络存储语义、后台任务和安全边界从第一天成为核心。

### Electron 桌面端

生态成熟、Node 能力直接，但安装体积和资源开销通常更高。若关键阅读库在 Tauri WebView 上存在无法解决的兼容问题，可重新评估。

### 直接 fork BookLore/Readest

能快速获得大量功能，但会继承 AGPL、既有领域模型、技术升级和上游合并成本。当前需求尚未清晰到足以选择 fork。

## 后果

- 首版单机体验和本地隐私更好。
- 需要处理 WebView2 等平台运行时差异。
- 手机原生端和浏览器远程访问不会自动获得。
- 如果用户确认首要场景是 NAS 多用户，本 ADR 应标记 Rejected，并在编码前另写 Web-first ADR。
