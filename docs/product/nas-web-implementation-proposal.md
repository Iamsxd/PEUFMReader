# PEUFMReader NAS 多用户 Web 实现方案

> 状态：产品方向与目标环境已确认，已授权开始实现。当前完成 M0 技术基线与 M1 导入分类闭环，具体技术选型仍可按验证结果调整。

## 1. 已确认的产品边界

- 产品部署在 NAS 上，是浏览器访问的多用户 Web 应用。
- 书库内容和元数据共享，阅读进度与阅读时长按用户隔离。
- 第一阶段只保证无 DRM 的 EPUB 和 PDF；其他格式分阶段增加，不包含 DRM 绕过。
- 自动分类首先支持作者、年份分组和固定题材分类，并允许可选 AI 辅助判断。
- 导入后默认复制到应用托管书库，不直接接管或改写来源文件。
- 首版阅读记录只包含进度和有效阅读时长。
- 部署目标是 Unraid Docker Compose，约 10 个用户、3000 本书，典型 PDF 约 20 MB（原始文件量级约 60 GB，另计封面、索引和数据库）。
- 首期只在局域网访问，未来可能开放公网；公网能力必须在安全加固里程碑完成后再启用。
- AI 同时支持本地 Ollama 与云端服务；Calibre 书库可以作为可选导入源，不作为运行时依赖。

## 2. 产品范围

### MVP 必须完成

1. 本地账号登录；管理员可以导入和整理，普通用户可以浏览和阅读。
2. 选择文件、拖放文件或监控一个导入目录。
3. 识别真实文件类型，计算哈希，隔离不支持或损坏的文件，并明确报告重复项。
4. 提取 EPUB/PDF 内嵌元数据、封面、目录、ISBN 和可用文本。
5. 通过可插拔提供者补全元数据；每个候选显示来源和置信度。
6. 按作者、年份动态分组，并把书籍映射到受控题材分类树。
7. 低置信度内容进入管理员“待整理”，可以接受、修改或忽略建议。
8. 内置 EPUB/PDF 阅读器；关闭后能可靠恢复当前用户的阅读位置。
9. 记录每位用户的开始/结束、有效阅读时长、进度变化和完成状态。
10. 来源文件不被静默覆盖；导入、重扫和分类操作可审计、可重试。
11. 元数据搜索、分类筛选、最近阅读和继续阅读。
12. 文件内容接口必须鉴权，用户不能绕过权限访问 NAS 绝对路径。

### 第二阶段

- 高亮、书签、笔记及导出。
- 全书库正文检索。
- MOBI/AZW3/FB2/CBZ/TXT 等格式。
- 扫描版 PDF 的可选 OCR。
- 批量元数据编辑、重复版本合并、分类规则编辑器。
- OPDS 只读目录。
- OIDC/LDAP、细粒度书库权限和审计后台。

### 第三阶段

- 离线客户端和多设备冲突处理。
- KOReader/Kobo 进度适配。
- 可选本地模型或云模型辅助摘要与问答。

### 明确不进入首版

- DRM 解密或绕过。
- 电子书编辑器和格式转换工厂。
- 社交、商店和推荐流。
- 微服务、Kubernetes、Redis 或独立消息队列。
- 原生 Windows/macOS/Android/iOS 客户端。

## 3. 推荐架构

### 技术栈

| 层 | 推荐 | 原因 |
|---|---|---|
| Web UI | React + TypeScript + Vite | 阅读器生态和组件测试成熟；可构建响应式/PWA 界面，无需首版引入 SSR |
| 服务端 | Go 模块化单体 | 对 NAS 资源友好；文件流、哈希、并发扫描和单二进制部署合适；最终选型仍可按团队经验调整 |
| API | REST/OpenAPI + SSE | REST 处理书库与进度；SSE 推送导入任务状态，首版无需完整 WebSocket 体系 |
| 数据库 | PostgreSQL | 多用户并发、后台任务抢占、迁移和后续全文索引比 NAS 场景下的 SQLite 更稳妥 |
| EPUB/更多流式格式 | 阅读器适配层；M0 使用 `epubjs` | `foliate-js` 缺少稳定 npm 发布，因此 M0 先隔离使用 `epubjs`；M1 前继续做兼容性与许可证 spike |
| PDF | PDF.js 或基于其封装的组件 | Apache-2.0，浏览器生态成熟，支持页渲染、搜索和文本层 |
| 后台任务 | 同一服务内 worker + PostgreSQL `jobs` 表 | 首版避免独立队列，同时支持崩溃恢复、重试和多实例互斥 |
| 元数据提供者 | Adapter/Plugin 接口 | Open Library、Google Books 或其他服务可独立启停、限流和替换 |
| AI 提供者 | Ollama/OpenAI-compatible adapter | 本地与云端可替换；AI 只给受控分类建议，不直接改写文件 |
| 部署 | Docker Compose：app + PostgreSQL | 适合通用 NAS；书库、数据库和缓存分别挂载持久卷 |

如果团队明显更熟悉 C#/.NET，可以把 Go 替换为 .NET；领域模型、API 和部署边界不需要改变。首版不建议照搬 BookLore 的 Spring Boot/MariaDB 组合，主要原因是小型 NAS 的资源与运维成本。

### 模块边界

```text
Browser UI / Reader
    |
Authenticated REST / SSE / HTTP Range
    |
Application services
    |-- Identity & Authorization
    |-- Import & Scan
    |-- Library & Metadata
    |-- Classification
    |-- Reading
    |-- Search
    `-- Integration ports
    |
Domain model
    |
Adapters: PostgreSQL / NAS Filesystem / Parser / Metadata & AI providers
```

采用模块化单体。模块通过应用服务和明确 DTO 交互，不让 React 直接访问数据库，也不让阅读器获得任何 NAS 绝对路径。

MVP 只设两种角色：

- `admin`：管理用户、导入文件、编辑元数据、处理分类建议。
- `reader`：浏览共享书库并维护自己的进度和阅读时长。

## 4. 受保护的文件存储与传输

### NAS 托管目录

```text
/data/inbox     # 可选监控来源，不直接删除
/data/staging   # 上传中的 .part 与解析临时文件
/data/library   # 校验后托管的不可变原始电子书
/data/cache     # 封面、缩略图、抽取文本等可再生数据
```

上传或监控文件先进入 staging，确认写入稳定、计算哈希并解析成功后再进入 library。同一文件系统使用原子移动；当 staging 与 library 是两个挂载点时，先复制到 library 所在卷的临时文件，再在目标卷内原子落盘。来源文件是否清理由管理员明确决定，应用默认不删除。

PostgreSQL 数据目录不能放在不可靠的 SMB/CIFS 共享挂载上，应使用数据库自己的本地持久卷。书库目录可以是 NAS 上的本地卷；如果应用跨机器访问网络共享，必须在 M0 专门验证锁、原子移动和断线行为。

### 鉴权文件流

PDF 和大型 EPUB 通过类似 `/api/book-files/{id}/content` 的鉴权接口读取。接口只接受数据库登记的逻辑 ID，校验当前用户权限，支持 Range 请求、ETag 和正确 MIME，并防止路径穿越。

Nginx/Caddy 可后续使用内部重定向优化大文件，但不能向浏览器暴露真实 NAS 路径。

## 5. 导入与自动分类流程

```text
发现文件
  -> 等待写入稳定
  -> MIME/魔数校验
  -> SHA-256 去重
  -> 解析内嵌元数据与封面
  -> ISBN/标题/作者候选匹配
  -> 外部元数据补全
  -> 作者/年份动态分组
  -> 固定题材分类映射
  -> 置信度门控
       |-- 高：自动采用，可撤销
       |-- 中：进入管理员待整理
       `-- 低：保持未分类
  -> 建立阅读资源与索引
```

### 分类信号优先级

1. 管理员锁定值和显式规则。
2. EPUB OPF、PDF Info、ISBN 等内嵌结构化字段。
3. ISBN 精确匹配的外部元数据。
4. 标题、作者、语言、出版年组合匹配。
5. 目录、简介和正文片段的确定性关键词/规则。
6. 可选 AI 建议。

AI 只允许返回受控分类 ID、自由标签建议、理由和置信度。输出使用 JSON Schema，并把可选分类限制为管理员定义的枚举；不能自行无限创建分类，也不能直接移动或修改文件。

每项建议保存来源、原值、归一化值、置信度、理由和 accepted/rejected 状态。阈值必须用用户的真实书籍样本校准。

## 6. 阅读记录设计

### 格式相关位置

- EPUB：保存 `{ href, cfi, chapterIndex, progression }`。
- PDF：保存 `{ pageIndex, yRatio, zoomMode }`。
- 所有格式额外保存 `overallProgress`，只用于列表展示和锚点失效后的回退。

不同文件版本的进度不直接共用锚点。EPUB 恢复顺序为 CFI → href/章节 → 整体比例。

### 用户隔离

- `reading_state` 唯一键为 `(user_id, book_file_id)`，保存最新位置和完成状态。
- `reading_session` 保存用户、起止时间、有效时长和起止位置。
- 翻页/章节切换触发防抖保存；标签页隐藏、会话失效或空闲超时关闭阅读会话。
- 只有页面可见且近期有阅读交互时累计有效时长。

所有浏览器都通过服务端读取 `reading_state`，所以用户登录后可以跨浏览器继续阅读；这不等于离线同步。

## 7. 核心领域模型

| 对象 | 含义 |
|---|---|
| User | 登录主体和阅读数据所有者 |
| Role | admin/reader 权限集合 |
| Work | 抽象作品，例如《三体》 |
| Edition | 某语言、出版社、ISBN、出版年的版本 |
| BookFile | 一个实际 EPUB/PDF 文件；同一 Edition 可以有多个格式 |
| Creator | 作者、译者、编辑等创作者及角色 |
| Category | 受控、可分层的固定题材分类 |
| MetadataCandidate | 带来源、分数和状态的元数据候选 |
| ClassificationDecision | 自动或人工分类决策及理由 |
| ReadingState | 某用户在某 BookFile 上的最新位置和完成状态 |
| ReadingSession | 某用户的一段有效阅读事件 |
| ImportJob | 可恢复、可重试的导入任务 |

`Work`、`Edition` 和 `BookFile` 不能合并成一张 books 表。共享书籍信息不能携带任何用户进度字段，避免多用户数据串写。

## 8. 里程碑与验收门槛

### M0：技术风险验证

- 用中文/英文 EPUB、复杂排版 EPUB、文本 PDF、扫描 PDF、超大 PDF和损坏文件建立测试语料。
- 验证鉴权 HTTP Range、PDF.js、foliate-js、中文字体、目录和位置恢复。
- 验证 Docker Compose、PostgreSQL 迁移和目标 NAS 的卷权限。

退出条件：两种核心格式可在局域网浏览器打开；十次关闭重开均恢复到可接受位置；损坏文件不会使服务崩溃；越权文件请求被拒绝。

### M1：书库与分类闭环

- 完成托管复制、去重、元数据、封面、候选合并、作者/年份分组、固定题材分类和管理员待整理。

退出条件：重复导入不产生重复记录；中断后任务可恢复；任何自动分类都可解释和撤销；托管文件哈希保持不变。

### M2：多用户阅读闭环

- EPUB/PDF 阅读器、继续阅读、进度保存、有效时长会话、完成状态。

退出条件：两名用户的进度与时长完全隔离；同一本书的 EPUB/PDF 位置互不污染；统计来自可追踪会话。

### M3：管理增强

- 高亮、批注、全文索引、分类规则编辑器、OPDS 和外部设备互操作。

## 9. 测试策略

- 领域单元测试：元数据合并、分类规则、进度回退和会话时长。
- 解析契约测试：每种格式维护可合法再分发的固定样本。
- 数据库迁移测试：空库、上一版本库、失败回滚。
- 导入集成测试：重复、损坏、半写入、重命名、权限不足和程序中断。
- 权限集成测试：未登录、reader、admin、越权文件 ID 和会话失效。
- 阅读器端到端测试：打开、翻页、改变字体/缩放、退出和恢复。
- 路径安全测试：任何输入路径都不能突破 library root。
- 性能基线：按用户真实书库规模设定，不凭空承诺固定数量。

## 10. 动手前剩余确认项

1. NAS 是群晖、威联通、Unraid 还是普通 Linux？是否可以运行 Docker Compose？
2. 预计用户数、书库数量级、典型 PDF 大小，以及扫描 PDF 占比是多少？
3. 服务只在局域网使用，还是会暴露到公网？
4. AI 优先使用 NAS 上的 Ollama，还是允许调用云端 OpenAI-compatible 服务？书名、简介和正文片段能否离开本机？
5. 是否需要从现有 Calibre 书库导入，或兼容 KOReader/Kobo/OPDS？

用户明确要求“开始”后，先进入 M0 技术验证，不直接铺开全部功能。
