# PEUFMReader 实现方案（已被 NAS Web 方案取代）

> 状态：Superseded。用户已确认 NAS 多用户 Web 方向，请以 [NAS Web 实现方案](./nas-web-implementation-proposal.md) 为准。

## 1. 默认产品假设

在用户确认前，方案暂按以下假设设计：

- 首要场景是个人本地书库，离线可用、隐私优先。
- Windows 是首个验证平台，但代码不绑定 Windows，后续可发布 macOS/Linux。
- 第一阶段只保证无 DRM 的 EPUB 和 PDF；其他格式分阶段增加，不包含 DRM 绕过。
- “自动分类”包括元数据归一化、受控分类、自由标签和智能书架，不等同于自动移动文件。
- 首版不做多人账号、公共互联网服务和强一致跨设备同步。

若目标实际是家庭 NAS 上的多用户 Web 服务，技术边界会明显变化，必须先重新确认 ADR-0001。

## 2. 产品范围

### MVP 必须完成

1. 选择文件、拖放文件或监控一个导入目录。
2. 识别真实文件类型，计算哈希，拒绝不支持/损坏文件，并对重复项给出明确结果。
3. 提取 EPUB/PDF 内嵌元数据、封面、目录、ISBN 和可用文本。
4. 通过可插拔提供者补全元数据；每个候选都显示来源和置信度。
5. 把书籍映射到受控分类树，同时允许自由标签和规则式智能书架。
6. 低置信度内容进入“待整理”，用户可以接受、修改或忽略建议。
7. 内置 EPUB/PDF 阅读器；关闭后能可靠恢复阅读位置。
8. 记录开始/结束、有效阅读时长、进度变化和完成状态。
9. 原始文件不被静默覆盖；导入、重扫和分类操作可审计、可重试。
10. 元数据搜索、分类/标签筛选、最近阅读和继续阅读。

### 第二阶段

- 高亮、书签、笔记及导出。
- 全书库正文检索（SQLite FTS5）。
- MOBI/AZW3/FB2/CBZ/TXT 等格式。
- 扫描版 PDF 的可选 OCR。
- 批量元数据编辑、重复版本合并、分类规则编辑器。
- OPDS 只读目录。

### 第三阶段

- 可选同步服务、多设备冲突处理。
- KOReader/Kobo 进度适配。
- 多用户/家庭书库或自托管 Web 模式。
- 可选本地模型/云模型辅助分类与摘要。

### 明确不进入首版

- DRM 解密或绕过。
- 电子书编辑器和格式转换工厂。
- 社交、商店、推荐流。
- 微服务、Kubernetes、独立消息队列。

## 3. 推荐架构

### 技术栈

| 层 | 推荐 | 原因 |
|---|---|---|
| 桌面壳 | Tauri v2 | 体积和内存通常低于 Electron；具备文件系统、系统协议和跨平台能力 |
| UI | React + TypeScript + Vite | 阅读器生态和组件测试成熟；桌面离线应用不需要 Next.js 的服务端复杂度 |
| 本地应用层 | Rust | 文件监控、哈希、任务调度、SQLite、流式读取和原生打包更稳妥 |
| 数据库 | SQLite（WAL）+ FTS5 | 单机零运维、事务可靠；足以支撑个人大型书库和后续全文索引 |
| EPUB/更多流式格式 | `foliate-js` 适配层 | Readest 已验证其跨平台格式能力；必须先做兼容性与许可证 spike |
| PDF | PDF.js 或基于其封装的组件 | Apache-2.0，浏览器生态成熟，支持页渲染、搜索和文本层 |
| 后台任务 | 进程内 worker + 持久化 `jobs` 表 | 首版避免额外服务，同时支持崩溃恢复、重试和可观测状态 |
| 元数据提供者 | Adapter/Plugin 接口 | Open Library、Google Books 或其他服务可独立启停、限流和替换 |

### 模块边界

```text
UI / Reader
    |
Tauri commands + 安全的本地资源协议
    |
Application services
    |-- Import & Scan
    |-- Library & Metadata
    |-- Classification
    |-- Reading & Annotation
    |-- Search
    `-- Sync ports (首版不实现远端 adapter)
    |
Domain model
    |
Adapters: SQLite / Filesystem / Parser / Metadata provider
```

采用模块化单体。模块通过应用服务和明确 DTO 交互，不让 React 直接执行 SQL，也不让阅读器直接拿任意系统路径。

### 本地文件服务

PDF 和大型 EPUB 不应通过 Tauri IPC 一次性复制成大块字节。应用提供受限的自定义协议，只允许访问数据库登记过的 `book_file_id`，支持 Range 请求，并校验路径穿越和 MIME。阅读器只持有逻辑 ID，不持有任意绝对路径。

## 4. 导入与自动分类流程

```text
发现文件
  -> 等待文件写入稳定
  -> MIME/魔数校验
  -> SHA-256 去重
  -> 解析内嵌元数据与封面
  -> ISBN/标题/作者候选匹配
  -> 外部元数据补全
  -> 分类体系归一化
  -> 置信度门控
       |-- 高：自动采用，可撤销
       |-- 中：进入待整理，推荐候选
       `-- 低：保持未分类
  -> 建立索引和阅读资源
```

### 分类信号优先级

1. 用户锁定值和用户显式规则。
2. EPUB OPF/PDF Info/ISBN 等内嵌结构化字段。
3. ISBN 精确匹配的外部元数据。
4. 标题、作者、语言、出版年组合匹配。
5. 目录、简介和正文片段的确定性关键词/规则。
6. 可选统计模型或 LLM 建议。

LLM 只返回受控分类 ID、自由标签建议、理由和置信度；不能直接创建无限新分类，也不能在无复核情况下修改原文件。

### 可解释性与回滚

每次建议至少保存：

- `target_field`：被建议修改的字段。
- `source_type/source_id`：内嵌、外部提供者、用户规则、内容分类器等。
- `raw_value/normalized_value`：原值和归一化结果。
- `confidence`：0 到 1。
- `reason`：命中的 ISBN、规则、关键词或模型理由。
- `status`：pending/accepted/rejected/superseded。
- `decided_by/decided_at`：人工或系统决策记录。

初始阈值可以设为 `>= 0.90` 自动采用、`0.65–0.90` 人工确认、`< 0.65` 保持未分类，但必须以真实样本校准，不写死为产品规则。

## 5. 阅读记录设计

### 位置不是一个百分比

- EPUB：保存 `{ href, cfi, chapterIndex, progression }`。
- PDF：保存 `{ pageIndex, yRatio, zoomMode }`。
- 所有格式额外保存 `overallProgress`，只用于列表展示和锚点失效后的回退。

EPUB CFI 可能因文件版本变化而失效，所以恢复顺序应是 CFI → href/章节 → 整体比例。不同文件版本的进度不直接共用锚点。

### 状态与事件分开

- `reading_state` 保存“当前读到哪里”，用于快速恢复。
- `reading_session` 保存“何时读了多久、从哪里到哪里”，用于统计和审计。
- 翻页/章节切换触发防抖保存；窗口隐藏、退出和空闲超时关闭会话。
- 只有窗口可见且近期有阅读交互时累计有效时长，避免一本书开一夜被计入统计。

未来同步时，以会话/注释事件作为事实，以 `reading_state` 作为可重建快照；这比只同步一个百分比更容易处理冲突。

## 6. 核心领域模型

| 对象 | 含义 |
|---|---|
| Work | 抽象作品，例如《三体》 |
| Edition | 某语言、出版社、ISBN、出版年的版本 |
| BookFile | 一个实际 EPUB/PDF 文件；同一 Edition 可以有多个格式 |
| Creator | 作者、译者、编辑等创作者及角色 |
| Category | 受控、可分层的分类体系 |
| Tag | 用户自由标签，不承担唯一分类职责 |
| SmartShelf | 保存的查询规则，不复制书籍数据 |
| MetadataCandidate | 带来源、分数和状态的元数据候选 |
| ClassificationDecision | 自动或人工分类决策及理由 |
| ReaderProfile | 阅读状态所有者；首版只有默认本地 profile |
| ReadingState | 某 profile 在某 BookFile 上的最新位置和完成状态 |
| ReadingSession | 一段有效阅读事件 |
| Annotation | 高亮、批注、书签的统一锚点模型 |
| ImportJob | 可恢复、可重试的导入任务 |

关键约束：`Work`、`Edition` 和 `BookFile` 不能合并成一张“books”表，否则重复格式、再版、进度锚点和元数据冲突会很快失控。

## 7. 里程碑与验收门槛

### M0：技术风险验证

- 用真实中文/英文 EPUB、复杂排版 EPUB、文本 PDF、扫描 PDF、超大 PDF、损坏文件建立测试语料。
- 验证 Tauri 自定义协议、Range 读取、PDF.js、foliate-js、中文字体、目录、搜索和位置恢复。
- 验证 Windows 安装包和数据库迁移。

退出条件：两种核心格式可以离线打开；十次关闭重开均恢复到可接受位置；损坏文件不会使应用崩溃。

### M1：书库与分类闭环

- 完成导入任务、去重、内嵌元数据、封面、候选合并、受控分类、待整理队列、搜索筛选。
- 支持 managed-copy 和 linked 两种库模式。

退出条件：重复导入不产生重复记录；中断后任务可恢复；任何自动分类都可解释和撤销；原文件哈希不变。

### M2：阅读记录闭环

- EPUB/PDF 阅读器、继续阅读、状态保存、有效时长会话、完成/重读状态。
- 对异常退出、文件移动和版本替换有明确降级行为。

退出条件：阅读位置和时长测试通过；同一本书的 EPUB/PDF 位置互不污染；书库页统计来自可追踪事件。

### M3：知识沉淀能力

- 高亮、批注、书签、导出、全文索引、分类规则编辑器。

### M4：互操作与同步

- OPDS、KOReader/Kobo 适配；根据真实需求决定是否引入同步服务和多用户。

## 8. 测试策略

- 领域单元测试：元数据合并、分类规则、进度回退和会话时长。
- 解析契约测试：每种格式维护可合法再分发的固定样本。
- 数据库迁移测试：空库、上一版本库、失败回滚。
- 导入集成测试：重复、损坏、半写入、重命名、移动、权限不足、程序中断。
- 阅读器端到端测试：打开、翻页、改变字体/缩放、退出、恢复、完成。
- 属性测试：任意路径和文件名不能突破 library root；任意重复任务不会重复落库。
- 性能基线：用用户真实书库规模确定，不凭空承诺固定数量。

## 9. 动手前必须确认的问题

1. 目标到底是“个人桌面软件”还是“部署在 NAS/服务器上的多用户 Web 应用”？这是架构分叉点。
2. 首发平台是否只要求 Windows？是否需要 Android/iOS 在第一年内加入？
3. 自动分类指哪一种：按作者/格式/年份自动分组，映射到固定学科/题材，还是读正文后由 AI 判断？
4. 导入后希望应用复制并管理文件，还是保持原目录不动、只建立索引？
5. “阅读记录”最低要求是进度与时长，还是首版就必须有高亮、笔记、年度统计和多端同步？
6. 预计书库规模、典型 PDF 大小，以及扫描 PDF 占比是多少？
7. 是否允许调用 Google Books/Open Library 等公网服务？敏感书名和正文是否必须永不离开本机？
8. 是否要求兼容现有 Calibre、KOReader、Kobo 或 OPDS 书库？

确认以上问题和 Proposed ADR 后，才进入 M0 技术验证，不直接开始完整功能开发。
